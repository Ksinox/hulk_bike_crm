# Расчётный период v0.7 — плавное переключение

Что проверять руками на preview перед мерджем в `main`. Юнит-тесты
(55 шт: 33 web + 22 api) уже покрывают всю математику резолвера —
здесь только сценарии, которые юнит-тестами не дотягиваются.

## Подготовка

1. `git checkout feature/billing-period-transition && git push -u origin feature/billing-period-transition`
2. В Dokploy → `hulk-web-preview` и `hulk-api-preview` переключить
   branch на `feature/billing-period-transition` → передеплоить
   (или `python scripts/redeploy_preview.py`).
3. Открыть `https://crm-preview.104-128-128-96.sslip.io`.
4. Залогиниться как директор или создатель.

## Шаг 1 — состояние «всё по-старому» (правило = 15)

В preview после первого деплоя должно стоять `rule_start_day = 15`
(seed-якорь от 2024-01-01, regular). Проверяем что ничего не сломалось:

- [ ] **Дашборд → Выручка**: лейбл `XX мес — YY мес` (без «переходный»),
      сумма не нулевая (или нулевая если БД свежая).
- [ ] **/rentals → KPI «За период»**: тот же лейбл, та же сумма что
      на дашборде.
- [ ] **/rentals → Архив**: видны завершённые аренды из прошлого
      периода (если есть).
- [ ] **/settings**: блок «День начала расчётного периода» показывает:
      *Сейчас: 15 · 15 мес — 14 след.мес*. История переключений
      скрыта (только один anchor — её не показываем).

## Шаг 2 — переключение 15 → 1

- [ ] В /settings ввести `1` → появляется превью раскладки
      (текущий, переходный, новая схема) синим блоком.
- [ ] Нажать «Сохранить» → toast «Переключение запланировано» с
      датами раскладки.
- [ ] Появилась таблица «История переключений» с двумя строками:
      сегодняшний transition и исходный regular от 2024-01-01.

## Шаг 3 — после переключения, идёт transition

Считаем что переключили 14 мая → transition `15 май → 31 май`.
Если переключили в другой день — даты в чеклисте сдвинутся.

- [ ] **/settings**: текущее правило = 1, период подкрашен
      жёлтым (transition), баннер «Идёт переходный период».
      Поле ввода disabled, кнопка тоже.
- [ ] Попробовать ввести любое значение и нажать «Сохранить» (через
      DevTools убрать disabled). Сервер должен ответить 400
      `transition_in_progress` — toast с ошибкой.
- [ ] **Дашборд → Выручка**: лейбл стал `15 май — 31 май (переходный)`.
      Сумма = только платежи 15–31 мая.
- [ ] **/rentals → KPI «За период»**: тот же лейбл, та же сумма.
- [ ] **Дашборд → график «Месяц»**: столбики идут с 15 по 31 мая
      (17 дней, не 30).
- [ ] Создать новую аренду + платёж сегодняшним числом → платёж
      попадает в transition (виден в сумме «За период»).
- [ ] **/rentals → Архив**: завершённые аренды с end_actual_at до
      15 мая → должны уехать в архив в течение ~часа (scheduler
      тикает раз в час; можно подождать или дёрнуть `pnpm --filter api dev`
      в отладке).
- [ ] **/whats-new (activity log)**: запись «Расчётный период 15 → 1.
      Переходный: 2026-05-15 — 2026-05-31. С 2026-06-01 — новая схема».

## Шаг 4 — после окончания transition (имитация: правка БД)

Чтобы не ждать до 1 июня — в preview-БД через `pnpm db:studio`
сдвинуть `transition_end_date` назад в прошлое (например на вчера).
Затем рефреш CRM.

- [ ] **/settings**: жёлтая подсветка ушла, поле снова активно,
      кнопка работает, баннер про переходный пропал.
- [ ] **Дашборд**: лейбл стал `01 <текущ.месяц> — <last> <текущ.месяц>`,
      kind=regular.
- [ ] Историю переключений всё ещё видно.

## Шаг 5 — регрессия «исторические аренды не пересчитываются»

- [ ] В /rentals → DateRangeFilter поставить старую дату (например
      март 2026) → отображаются аренды из того периода как раньше.
- [ ] Если есть аренда из марта 2026, её сумма не изменилась после
      переключения 15→1.
- [ ] **Дашборд** при выборе «За месяц» в режиме `month` график
      показывает текущий период (transition или новый regular).
      Исторические периоды через другой UI пока не доступны — это
      существующая семантика.

## Шаг 6 — проверка повторного переключения после окончания transition

- [ ] В /settings выставить 1 → 10 (или любое другое отличное от 1).
- [ ] Превью считает новый transition корректно.
- [ ] Сохранить → 3 anchor'а в истории.

## Если что-то идёт не так

- В preview-БД таблица `billing_period_anchors` — можно посмотреть
  через `pnpm --filter api db:studio` или прямой SQL.
- Чтобы откатить всю историю: `DELETE FROM billing_period_anchors`,
  миграция при следующем рестарте API пересоздаст seed-якорь.
- Активность в `activity_log` с `action='billing_period_switch'`
  и `action='auto_archive_period'`.
