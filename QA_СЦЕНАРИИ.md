# Сценарии использования CRM «Халк Байк» — журнал QA-прогона

**Назначение.** Полный каталог пользовательских сценариев (клиенты, скутеры, заявки,
аренды и все их вариации) + журнал прохождения. Написан с позиции владельца бизнеса
аренды скутеров: не только «нажми кнопку», а «какая реальная ситуация возникает за
прилавком и умеет ли CRM её закрыть».

**Кто и зачем читает.** Любой агент (или человек) открывает файл и сразу видит: какой
пункт когда и на какой сборке проверен, что сломано, чего вообще нет в системе.
Прогон ведём прямо здесь — правим статус-строку каждого сценария.

23. **[ТРЕБОВАНИЕ владельца 2026-05-29] Режим «по сумме клиента» должен предлагать
    самый ВЫГОДНЫЙ для клиента вариант (справедливо).** Сейчас по умолчанию ставится
    самый дорогой тариф «1-2 дня» (1300) → клиент получает минимум дней. Нужно
    автоматически подбирать тариф, дающий клиенту **максимум дней за его деньги**.
    ⚠️ **Нюанс корректности (важно для реализации):** нельзя просто взять самую низкую
    ставку — тариф ДОЛЖЕН соответствовать получившемуся числу дней (диапазону тарифа):
    - тарифы по сроку: 1-2 дня 1300, 3-6 дней 700/600, 7-29 дней 500, 30+ дней 400 ₽/сут;
    - нельзя дать ставку «30+ дней» (400) за 8 дней — 8 не входит в диапазон 30+;
    - посуточная экипировка входит в цену дня (тариф + экип/сут), и влияет на число дней.

### Как отмечать сценарий
У каждого сценария есть строка статуса. После проверки агент переписывает её:

```
📋 ⬜ не проверено | дата: — | сборка: — | агент: — | заметка: —
```
→ становится, например:
```
📋 ✅ OK | дата: 2026-05-28 | сборка: v0.7.5 | агент: claude | заметка: всё ок
```

**Иконки статуса:**
- ⬜ **не проверено** — ещё не прогоняли (или сброшено новой сборкой)
- ✅ **OK** — работает как ожидается
- ❌ **BUG** — баг (описать в «заметка», при возможности — что чинить/где)
- ⚠️ **GAP** — функции в CRM нет / действие не продумано (это и есть «дыры» владельца)
- ⏭️ **N/A** — неприменимо в текущей модели бизнеса

### Правило перепроверки при новой сборке
Когда поднимается новая сборка (например `v0.7.6`):
1. Обновить «Текущая проверяемая сборка» вверху.
2. Все пункты со старой сборкой считаются **устаревшими** — их нужно прогнать заново.
   Можно не стирать прошлый результат, а дописать новую строку под старой
   (история прогонов копится). Минимально — поменять иконку на ⬜ у затронутых.
3. ⚠️ GAP-пункты при новой сборке проверяются особо: возможно, дыру уже закрыли.

### Действия по аренде (из `RentalCard.tsx`)
- **new_request:** Назначить встречу (`schedule`), Отменить (`cancel`)
- **meeting:** Выдать скутер (`activate`), Отменить (`cancel`)
- **active / overdue:** Завершить аренду (`complete`) + extras
- **returning:** Завершить аренду (`complete`), Отменить возврат (`cancel-return`) + extras
- **completed:** Перевести в активную (`revert-completion`)
- **completed_damage / problem:** Сбросить «проблемная» (`normalize-status`), Возобновить аренду (`resume-damage`), Претензия (`claim`) + extras
- **police / court:** Скутер вернулся — завершить (`complete`), Закрыть с ущербом (`set-damage`), Отменить дело (`revert-police`) + extras
- **extras (живые статусы):** Зафиксировать/Изменить ущерб (`set-damage`), Начислить долг (`charge-debt`), Сбросить просрочку (`forgive-overdue`), Не выходит на связь / Снять (`mark-/unmark-unreachable`)
- **footer/CTA:** Принять оплату (`PaymentAcceptDialog`), Продлить (`ExtendRentalDialog`), Заменить скутер (`SwapScooterDialog`), Изменить экипировку (`EquipmentChangeDialog`), Пополнить залог (`SecurityTopupDialog`), Редактировать (`RentalEditModal`)
