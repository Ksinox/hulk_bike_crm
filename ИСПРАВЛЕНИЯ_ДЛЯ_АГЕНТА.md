**Как правильно (по словам владельца):**
«Принять платёж» = ОДНО окно, payment-first. Внутри оператор выбирает ЧТО оплатить/сделать
(комбинируемо): **продлить аренду** (только если аренда активная) · **пополнить залог** ·
**оплата ущерба** · **просрочка/штрафы** · **ручной долг / неоплаченная аренда / паркинг**.
Комбинации: «продлить + залог», «только залог», «только ущерб» и т.д.
«Завершить аренду» — ОТДЕЛЬНОЕ окно («прощаемся»), внутри — возможность зачесть залог, если у
клиента нет денег. **Не сливать завершение в «Принять платёж».**

**Что сделать:**
1. Заголовок `PaymentAcceptDialog` → всегда **«Принять платёж · #NNNN»** (подзаголовок адаптивный:
   «есть долг N ₽…» / «нет долга — можно продлить / пополнить залог»). Никогда не «Продление аренды».
2. Блок «Продление» (Step 2) показывать **только для активной аренды**. Для completed/
   completed_damage/cancelled — продление скрыто, показываются только долг/залог/ущерб.
3. Баннер ущерба «Внести платёж» / «Внести платёж по ущербу» → открывать **то же** `PaymentAcceptDialog`
   (со скроллом/фокусом на строке «Ущерб по акту»), НЕ `DamageReportPaymentDialog`. После паритета —
   `DamageReportPaymentDialog` удалить (и его вызовы).
4. И кнопка в шапке «Принять оплату», и баннер ущерба → одна точка входа (одно окно).
- ⚠️ НЕ трогать `distribute()` / forgive / `extendInplaceAsync` — это UX-консолидация входов +
  заголовок + условный показ продления. Регресс-проверить 4 случая: активная (продление+долг+залог),
  просрочка (прощение), завершённая-с-ущербом (только долг), паркинг.

### R2. Публичная анкета (заявка) — развить

**🎯 ЦЕЛЕВОЙ СКВОЗНОЙ СЦЕНАРИЙ (то, ради чего всё это):**
`Заявка (клиент сам) → Клиент создан → сразу предложение создать аренду → аренда
преднастроена данными из заявки → оператор выбирает конкретный скутер → выдача + договор.`

Подробно, шаг за шагом:
1. Клиент по ссылке заполняет анкету: контакты, паспорт, фото, **+ экипировка + модель +
   период** (новое, см. ниже).
2. Заявка приходит в CRM. Оператор жмёт «Оформить сейчас» → **создаётся клиент** (с фото в
   правильных слотах — см. F18).
3. **СРАЗУ после сохранения клиента система предлагает «Создать аренду с этим клиентом»** и
   открывает панель «Новая аренда», где **уже подставлено из заявки:** клиент, **модель**,
   **экипировка**, **период (дни)**. Оператору остаётся:
   - **выбрать конкретный свободный скутер** этой модели,
   - подтвердить залог/оплату старта (наличными — см. R2.6),
   - **«Создать и выдать» → распечатать договор + акт выдачи** (шаблон «Договор + акт»).
4. Всё, что клиент выбрал в анкете, должно **точно стыковаться** (match) на каждом шаге:
   анкета → клиент → панель аренды → договор.
