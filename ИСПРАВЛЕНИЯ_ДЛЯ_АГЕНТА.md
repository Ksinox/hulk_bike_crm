## 🔴 P0 — Критично (деньги / целостность данных)

### F1. Долг по ущербу не отображается на самой аренде
- **Сейчас:** при завершении аренды «с ущербом» остаток долга (напр. 1 500 ₽ после зачёта
  залога) **не виден на карточке аренды** — она закрывается как «Завершена», ДОЛГ 0. Долг
  при этом реально существует и виден на карточке **клиента** («Долг по ущербу: N ₽ по всем актам»).
- **Надо:** аренда с непогашенным ущербом → статус «с ущербом» (не «Завершена») + показывать
  остаток долга. Долг аренды и долг клиента должны быть согласованы.
- ⚠️ (перепроверить) — платежи/ущерб менялись (v0.8.29–0.8.33), убедись что не закрыто частично.

### F2. Завершение «с ущербом» должно переиспользовать «хороший» диалог ущерба
- **Контекст:** есть два диалога фиксации ущерба. «Зафиксировать ущерб» на **активной** аренде —
  полноценный: редактируемые суммы позиций, зачёт залога, «к доплате», тумблер «отправить в
  ремонт». Поток **«Завершить → Есть ущерб»** ранее использовал урезанный список (суммы не
  редактируются, авто-ремонт, долг терялся).
- **Надо:** в завершении аренды переиспользовать ИМЕННО полноценный диалог. Тогда чинятся:
  редактируемые суммы, выбор «в ремонт / обратно в работу» (не авто), и F1 (долг не теряется).
- ⚠️ (перепроверить на v0.8.33).

### F3. Перекрытие периодов паркинга не блокируется  ← подтверждено владельцем
- **Сейчас:** один и тот же день можно отметить паркингом дважды (новый период пересекается с
  существующим) — система не запрещает, и **дата возврата сдвигается повторно** (двойной учёт дня).
- **Надо:** запретить пересечение периодов паркинга (день не может попасть в паркинг дважды).

### F4. KPI на странице «Аренды» не считал просрочки/долг
- **Сейчас (исторически):** верхняя KPI-полоса стр. «Аренды» показывала «Просрочек 0 / Долг 0 ₽»
  при реальной просроченной аренде с долгом; главный дашборд считал верно.
- **Надо:** KPI стр. «Аренды» = главный дашборд.
- ⚠️ (перепроверить) — стр. «Аренды» переработана (новый KPI-бар + вкладки); проверь при наличии
  просроченной аренды, если расхождение осталось — исправь.

## 🟢 Требования / новый функционал

### R1. Ревизорская история (Хронология событий) — доработать UX
Лента есть (фильтры Все/Продления/Замены/Долги), но строки **не кликабельны**, **мало данных**,
**нет группировки**, drawer ломкий, иконки одинаковы. Сделать:
1. **Строка кликабельна** → деталь-панель/модалка: что было/стало, связанные сущности
   (платёж/акт/связка-продление), автор, время, кнопки «Открыть связанное» и «Отменить действие».
2. **2-уровневые строки** (действие+дата сверху, сводка снизу). Что показывать в сводке по типам:
   - Продление: «04.06 → 11.06, +7 дн × 500 = 3 500 ₽ · нал · принял Руслан»
   - Замена скутера: «Jog #03 → Jog #04 · причина: … · старый → ремонт · Директор»
   - Платёж: «Паркинг 250 + ущерб 1500 = 1750 ₽ · нал · долг 1750 → 0»
   - Ущерб: позиции (фара 2500 / прокол 1000), итог, зачёт залога, остаток, автор
   - Заметка: «было / стало» при изменении
   - Паркинг: даты + формула «3 дн × 250 − 1 беспл = 500 ₽», включён ли free-день
3. **Группировка по дням** (Сегодня / Вчера / 28 мая…).
4. **Иконки + цвет по типу** (✏️ заметка, 🅿️ паркинг, 💰 платёж, 🔧 ущерб, 🔁 замена); денежный итог справа.
5. **Стабильный drawer** (не закрывать на случайные клики).
6. **Поиск/фильтр по автору и сумме**; **экспорт** ленты в CSV/PDF.

### R2. Публичная анкета (заявка) — развить

**🎯 ЦЕЛЕВОЙ СКВОЗНОЙ СЦЕНАРИЙ (то, ради чего всё это):**
`Заявка (клиент сам) → Клиент создан → сразу предложение создать аренду → аренда
преднастроена данными из заявки → оператор выбирает конкретный скутер → выдача + договор.`

Подробно, шаг за шагом:
1. Клиент по ссылке заполняет анкету: контакты, паспорт, фото, **+ экипировка + модель +
   период** (новое, см. ниже).
2. Заявка приходит в CRM. Оператор жмёт «Оформить сейчас» → **создаётся клиент** (с фото в
   правильных слотах — см. F18).
3. **СРАЗУ после сохранения клиента система предлагает «Создать аренду с этим клиентом»** и
   открывает панель «Новая аренда», где **уже подставлено из заявки:** клиент, **модель**,
   **экипировка**, **период (дни)**. Оператору остаётся:
   - **выбрать конкретный свободный скутер** этой модели,
   - подтвердить залог/оплату старта (наличными — см. R2.6),
   - **«Создать и выдать» → распечатать договор + акт выдачи** (шаблон «Договор + акт»).
4. Всё, что клиент выбрал в анкете, должно **точно стыковаться** (match) на каждом шаге:
   анкета → клиент → панель аренды → договор.

**Детали по пунктам анкеты:**
Клиент по ссылке заполняет параметры аренды; при конвертации открывается преднастроенная панель «Новая аренда».
1. **Выбор экипировки** (из каталога, несколько позиций) → подтягивается в аренду.
2. **Выбор модели** (Jog/Gear/Honda/Tank) → при сохранении конвертирует клиента и **сразу открывает
   панель «Новая аренда» с предзаполненной моделью** (оператор выбирает только конкретный свободный скутер).
3. **Выбор периода** (дни) + **встроенный калькулятор** (тариф модели × дни + экипировка/сут =
   ориентировочная стоимость) → период подтягивается в дни аренды.
4. **Все данные «match»** анкета → клиент → панель аренды (модель, экипировка, период, контакты, паспорт, фото).
5. **Фото-шаги:** после загрузки фото — **предпросмотр** + кнопки **«Перефоткать»** и
   **«Подтвердить»** (засветы, смазанные права). Сейчас фото грузится без возможности проверить.
6. **Финальное информ-окно** перед отправкой: «**Внимание: старт аренды оплачивается наличными.
   Возьмите с собой наличные средства для оплаты.**» + кнопка «**Ок, возьму наличку**» (фирменный модал).
7. **Пользовательское соглашение** перед подачей (текст ниже). Кнопка «**Принять**» активна
   **только после прокрутки до конца текста**. Без принятия — отправка заблокирована.
